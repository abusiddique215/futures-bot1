# Plan 19 — Lux Bot (NQ 10m, Discord Signal Source) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development`.

**Goal:** Third NQ bot, distinct from SurgeBot + PropBot because its signal source is an external Discord channel rather than internal indicator logic (per the "RP - Lux Bot (Discord - NQ 10m)" label visible at ~18:30 in the VSL). Lux Bot listens to a Discord channel for trade signals, parses them into OrderIntents, runs them through the risk gate, and executes. After this plan: 5 of 6 user-facing bots deployable; the codebase has first-party signal ingestion infrastructure for any future Discord-based or webhook-based bots.

**Architecture:** Lux Bot is a `BotSpec` (Plan 12) wiring a new `DiscordSignalStrategy` (consumes signals from an injected `SignalSource` Protocol) + `CustomWindows` or `AlwaysOn` schedule (Discord posts can arrive any time) + `EFAStandardEoDDrawdown` for live-account safety + MarketSpec for NQ/MNQ.

`SignalSource` Protocol abstracts the actual input — production uses a `DiscordSignalSource` reading from a Discord bot's message queue; tests use a `FixtureSignalSource` reading from a JSON file. Both implement `async iter_signals() -> AsyncIterator[SignalEvent]`.

**Tech Stack:** New dep: `discord.py` (the canonical Python Discord library — well-maintained, async-native). Pin to a recent stable version (`discord.py>=2.4,<3.0`).

**VSL fidelity (observable constraints honored):**
- **Market**: NQ (per "Discord - NQ 10m" label)
- **Timeframe**: 10-minute bars
- **Discord source**: signals come from Discord — this is the load-bearing differentiator vs SurgeBot/PropBot
- **24/7 framing implied**: Lux Bot is in the maintenance family; Discord posts can arrive overnight

**Important distinction (not the $99/mo community product):** Lux Bot's Discord source is an INPUT to our bot, not the OUTPUT product. The $99/mo Profit Insider Discord community is a separate product we are not cloning. Lux Bot just happens to take its signals from a Discord channel — likely a "signal channel" that a human trader populates or that another bot of theirs posts to. The user will configure the channel ID; we do not connect to the public Profit Insider Discord.

**Signal format assumption:** The VSL doesn't show the actual signal message format. We design a simple structured format the user can configure to match whatever their signal channel uses. Example signal messages: `BUY NQ @20100 SL=20070 TP=20160` or `LONG MNQH26 limit 20100 stop 20070 target 20160`. The parser is regex-based and tolerant of common variations; unparseable messages are logged + ignored (never crash on a typo).

## Tasks

### T1: `SignalEvent` + `SignalSource` Protocol

`src/bot/signals/source.py`. Frozen dataclass `SignalEvent(received_at, symbol, side, qty, limit_price, stop_loss, take_profit, raw_text, source_id)`. Protocol `SignalSource.iter_signals() -> AsyncIterator[SignalEvent]`. Module-level docstring documents the signal format expected.

Tests:
- SignalEvent is frozen.
- Protocol can be satisfied by FixtureSignalSource (compile-only check).

Commit: `feat(signals): SignalEvent dataclass + SignalSource Protocol`.

### T2: Signal message parser

`src/bot/signals/parser.py`. Function `parse_signal_message(text: str, default_symbol: str) -> SignalEvent | None`. Regex-based; tolerant of:
- `BUY NQ @20100 SL=20070 TP=20160`
- `LONG MNQH26 limit 20100 stop 20070 target 20160`
- `SHORT 1 NQ at 20100 stop 20130 tp 20040`
- Plus emoji-prefixed variants (signals channels often use 🟢/🔴).

Return None on unparseable input (logged at INFO, not WARN — typos are normal).

Tests:
- All 4 example formats parse correctly.
- Junk text returns None.
- Missing TP → TP defaults to None (downstream strategy decides).
- Case-insensitive (BUY/buy/Buy all work).
- Numeric variations (20,100 vs 20100 vs 20100.0).

Commit: `feat(signals): parser for common Discord signal message formats`.

### T3: `FixtureSignalSource` (tests)

`src/bot/signals/fixture_source.py`. `FixtureSignalSource(events: list[SignalEvent], emit_rate_seconds: float = 0.0)`. `iter_signals()` yields events with optional delay (default 0 = all at once).

Tests:
- Yields all events in order.
- Empty input completes cleanly.
- emit_rate respected (use `asyncio.wait_for` to verify timing).

Commit: `feat(signals): FixtureSignalSource for tests`.

### T4: `DiscordSignalSource` (production)

`src/bot/signals/discord_source.py`. `DiscordSignalSource(token, channel_ids, default_symbol)`. Uses discord.py — wraps it inside a `Bot` subclass with `on_message` handler. Each message in a watched channel → parse → enqueue → yield from `iter_signals`.

Tests: use `unittest.mock.AsyncMock` to fake the Discord client; assert messages flow from on_message → iter_signals queue.

CRITICAL safety: signals NEVER bypass the risk gate. Even if Discord goes berserk and posts 1000 messages, each generated OrderIntent goes through `TopstepRiskGate.approve_or_deny`. The parser's `qty` field is also capped at the bot's risk-policy `max_position` regardless of what the message claimed.

Tests:
- on_message in watched channel → SignalEvent yielded.
- on_message in unwatched channel → ignored.
- Malformed message → logged + ignored, no crash.
- Bot reconnect: events still flow.

Commit: `feat(signals): DiscordSignalSource (discord.py-backed)`.

### T5: `SignalStrategy`

`src/bot/strategy/signal_strategy.py`. Implements Strategy Protocol but the constructor takes a `SignalSource`. Strategy maintains an internal queue; `on_bar` checks for any signals received since last bar, converts to OrderIntents, applies a cap (max one new signal per bar to prevent flooding), and yields.

Tests:
- No signals → no intents emitted.
- 1 signal queued before bar → 1 intent on next bar.
- 5 signals queued before bar → 1 intent emitted, 4 retained in queue for next bar.
- Signal for wrong symbol → skipped.

Commit: `feat(strategy): SignalStrategy (consumes SignalSource → OrderIntent stream)`.

### T6: BotSpec YAML + registry

Registry: `register_strategy("signal_strategy", lambda p: SignalStrategy(...))` — the source needs to be injected externally since it requires runtime config (token, channels). Registry returns a partial; FleetRuntime completes the wiring when it sees `strategy_id: signal_strategy` (looks up `DiscordConfig` from runtime secrets).

`config/bots/lux_bot.yml`:
```yaml
name: lux_bot
enabled: false   # disabled by default — requires Discord token in secrets
symbol: MNQH26
strategy_id: signal_strategy
strategy_params:
  source_type: discord
  discord_channel_ids: []   # user fills in
  max_signals_per_bar: 1
  default_symbol: MNQH26
risk_policy: efa_standard
risk_params:
  mll_amount: 2000
schedule_type: always   # Discord posts arrive any time
journal_path: state/journal_lux_bot.db
```

Tests:
- YAML loads with placeholder channel_ids.
- Registry build succeeds even without a real Discord token (uses FixtureSignalSource fallback if env var `LUX_BOT_FIXTURE_PATH` is set).

Commit: `feat(config,registry): lux_bot.yml + signal_strategy registry entry`.

### T7: End-to-end integration test

`tests/integration/test_lux_bot_e2e.py`. Drives synthetic bars + FixtureSignalSource with 3 pre-parsed signal events → FleetRuntime → SimClient → Journal. Asserts:
- 3 signals → 3 OrderIntents → 3 approved orders → 3 fills.
- Risk gate denials still apply (signal claims qty=100 but risk policy caps at qty=5 → cap enforced).

Commit: `test(integration): Lux Bot end-to-end (signal stream → fleet → journal)`.
